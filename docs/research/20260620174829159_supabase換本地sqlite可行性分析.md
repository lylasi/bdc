# Supabase 換成「本地 SQLite」可行性分析

- 日期：2026-06-20
- 範圍：評估把目前的 Supabase 雲端同步，改成「本地 SQLite」是否可行
- 結論性質：架構可行性調研 + 替代方案建議

---

## 0. TL;DR（先看這段）

1. **「用瀏覽器內的本地 SQLite 取代 Supabase」是個偽命題、不可行。**
   本專案是純前端靜態 SPA，跑在瀏覽器沙箱裡。Supabase 在這裡的真正價值是
   **雲端跨裝置同步 + 身分登入 + Realtime**。本地 SQLite（不論 sql.js / wa-sqlite + OPFS）
   只是「又一個單機本機儲存」，而你**現在已經是 local-first**（資料早就在 localStorage + IndexedDB），
   再塞一個本機 SQLite，解決不了 Supabase 解決的任何問題（跨裝置、雲備份），純屬原地踏步。

2. **你的真實痛點大機率不是「額度太少」，而是「免費專案 7 天沒有資料庫活動就被自動暫停」。**
   本專案的遠端資料是「單使用者單列 JSON snapshot」，體積極小，幾乎不可能撞到 500MB / 流量上限。
   所以「最近一直被封」的成因，是 Supabase Free 的 **7 天閒置自動 pause**（恢復要約 30 秒冷啟動）。

3. **正解不是本地 SQLite，而是換一個「不會閒置暫停、額度更大」的同步後端。**
   而且因為目前同步接口面**非常窄**（只有 `get_snapshot` / `save_snapshot` 兩個 RPC + auth + realtime 訂閱），
   遷移成本集中在一兩個檔案，`sync-core.js` 的合併邏輯可 100% 複用。

4. **如果你心裡就是想要 SQLite —— 有個兩全方案：Cloudflare D1。**
   D1 本質就是 SQLite，免費層 5GB / 每日 500 萬列讀、10 萬列寫，**且不會因閒置暫停**。
   既滿足「我要 SQLite」的心理，又保留雲端跨裝置能力。對本專案的小資料量綽綽有餘。

## 1. 現況：Supabase 在架構中到底做了什麼

採用的是 **Scheme C：snapshot 同步**（見 `modules/sync-supabase.js` 抬頭註解）。

### 1.1 Supabase 同時承擔三件事

| 能力 | 用在哪 | 證據 |
| --- | --- | --- |
| **身分認證 Auth** | 登入/註冊/OTP/magic link/改密碼 | `features/sync/sync.js` 大量 `auth.signInWithPassword / signUp / signInWithOtp / resetPasswordForEmail / updateUser / getSession / onAuthStateChange` |
| **儲存 + 樂觀鎖** | 存整個 App 的 JSON 快照 | `modules/sync-supabase.js` 的 `pullSnapshot()` → RPC `get_snapshot()`；`pushSnapshot()` → RPC `save_snapshot(expected_version, p_payload)`，衝突碼 `40001` |
| **Realtime** | 訂閱 `snapshots` 表變更，推動跨端刷新 | `subscribeSnapshotChanges()` → `supabase.channel('snapshots-changes').on('postgres_changes', …)` |

### 1.2 資料模型極簡（這點對遷移很關鍵）

- 遠端只有一張 `snapshots` 表，**一個使用者一列**：`{ version, updated_at, payload }`。
- `payload` 是整個 App 打包成的一份 JSON：`vocabulary / articles / qa / dictation / assistant`
  （見 `modules/sync-core.js` 的 `buildLocalSnapshot()`）。
- 推送前還會 `compactAnalyzedArticles()` 壓縮（句析只留最新、片語只留 top3），刻意縮小體積。
- 合併採 **group-level LWW**（`lwwMerge()`，靠各分組的 `updatedAt` 比大小）。

> 換句話說：真正跟 Supabase 綁定的只有「pull/push 兩個 RPC + auth + realtime 訂閱」這層薄殼。
> `sync-core.js`（481 行的合併/套用/還原邏輯）**與後端無關**，換後端時可原封不動複用。

### 1.3 同步觸發頻率（為何會吃額度/一直有活動）

依 `docs/research/20260326230623669_自动同步触发操作清单.md`：

- **事件驅動**（非定時輪詢）：本地任何資料變更發 `bdc:data-changed` → `scheduleAutoSync()`。
- 防抖 6s、最小間隔 20s、realtime 事件節流 5s。
- 觸發面很廣：詞書 / 文章 / QA / assistant / 聽寫設定 / 批改歷史都會觸發，
  其中還有「觸發了但根本沒上雲」的無效觸發（articleMetas、批改歷史）。

**啟示**：頻繁的同步活動其實會「幫你保活」，反而較不會 7 天閒置暫停；
真正風險是你**有一段時間沒打開 App**，遠端就靜置滿 7 天被 pause。

## 2. 核心矛盾：為什麼「瀏覽器內本地 SQLite」不可行

這是純前端靜態 SPA，跑在瀏覽器裡。「本地 SQLite」在這個形態下只有兩條路，**兩條都不解決你的問題**：

### 路線 A：瀏覽器內 WASM SQLite（sql.js / wa-sqlite + OPFS / IndexedDB）

- 技術上**能跑**，但 SQLite 檔案最終還是存在「這一台瀏覽器的本機儲存」裡。
- 而本專案**已經是 local-first**：核心資料就在 `localStorage`，AI 衍生快取在 IndexedDB（`bdc-cache`）。
- 所以這等於：**把已經在本機的資料，換個格式再存一次到本機**。
  - ❌ 不能跨裝置（手機 ↔ 電腦不通）
  - ❌ 不能雲備份（換機/清快取就沒了）
  - ❌ 不能多端 Realtime
- 一句話：**為了不坐船過河，改成在原地挖了個游泳池。** Supabase 的價值一個都沒接住。

### 路線 B：真正的本地 SQLite 檔案（`app.db`）

- 瀏覽器**沒有權限**直接讀寫使用者磁碟上的 `.db` 檔（沙箱限制）。
- 要做到只能改變應用形態：
  - 套 **Tauri / Electron** 變桌面程式 → 失去「打開瀏覽器即用」，且跨裝置仍要另外想辦法。
  - 或 **自建後端**（Node + better-sqlite3 等）→ 那 SQLite 只是後端的儲存選型，
    你真正要做的是「自架一個一直在線的同步服務」，這跟「本地」恰恰相反。

> **結論**：在「保持純前端靜態 SPA」的前提下，「本地 SQLite 取代 Supabase」做不到 Supabase 在做的事。

## 3. 釐清你真正想要什麼（決策岔路口）

替代方案完全取決於一個問題：**你需不需要跨裝置同步？**

### 場景 ①：其實只在「單一裝置」用，雲端可有可無

→ 你根本不需要 SQLite，也不需要任何後端。**直接關掉雲同步**即可，零成本、零風險。

- 資料已經在 `localStorage` + IndexedDB，照常運作。
- 用既有的 `modules/local-backup.js`（`createBackup()` / `listBackups()` / `loadBackupPayload()`）
  做「手動匯出/匯入 JSON 快照」當備份。
- 改動：把 `ai-config.js` 的 `SUPABASE` 留空時，讓 `features/sync/sync.js` 直接停用同步入口即可
  （目前未設定就已大致降級，可再加一個明確的 disabled 狀態）。

### 場景 ②：需要手機 ↔ 電腦跨裝置同步（這才是 Supabase 的價值）

→ 本地 SQLite 給不了，必須有「雲端」。換一個**更穩、更省、不會閒置暫停**的後端。見第 4 節。

## 4. 針對真實痛點的可行替代方案（場景②）

因為同步接口面只有 `pull / push / auth / realtime` 四件事，遷移成本低。以下按推薦度排序。

### ⭐ 選項 1：Cloudflare D1（= SQLite）+ Cloudflare Worker（最推薦）

- **D1 本質就是 SQLite**，正面命中你「想用 SQLite」的訴求，同時保留雲端跨裝置。
- 免費層（2026）：**5GB 儲存、每日 500 萬列讀、10 萬列寫**，而且 **Workers 沒有閒置暫停機制**
  → 直接根治「7 天被 pause」。本專案單列 snapshot + 每天幾十次同步，用量連零頭都不到。
- 本專案已有 Cloudflare Worker 使用經驗（`docs/cf-worker-proxy.md`），技術棧熟悉。
- 工作量：寫一個小 Worker 暴露 `GET /snapshot`、`POST /snapshot`（帶 `expected_version` 樂觀鎖，
  對應 D1 的一張 `snapshots(user_id, version, updated_at, payload)` 表），
  前端改寫 `modules/sync-supabase.js` 的 `pullSnapshot/pushSnapshot`。
- 取捨：
  - **Auth** 要自理。單使用者場景可極簡化成「一組固定 token / 通行碼」放在 Worker 環境變數比對。
  - **Realtime** 免費層沒有現成 postgres_changes；降級成「輪詢 + 比對 version」即可
    （snapshot 模型本來就是拉取再合併，輪詢完全夠用）。

### 選項 2：PocketBase（自託管，內建 SQLite + Auth + Realtime + REST）

- 一個 Go 單檔執行檔，**內建 SQLite**，自帶 **Auth + Realtime + REST/管理後台**，
  幾乎是 Supabase 能力的一對一替身 —— 同時滿足「我要 SQLite」+「保留現有全部能力」。
- 放在任意小 VPS / 家用機 / NAS 上跑，**資料就是一個 SQLite 檔，自己掌控、可隨手備份**。
- 工作量：改寫 `sync-supabase.js`（換成 PocketBase SDK 或 REST），
  auth UI 可大致沿用（PocketBase 也有 email/password、OTP）。Realtime 用其 SDK 的 subscribe。
- 取捨：要自己有一台「一直在線」的機器（這是「本地/自託管」的代價）。

### 選項 3：其他

- **Turso**：託管版 libSQL（SQLite 衍生），也是「SQLite 系」，免費額度大；但同樣要自配 auth/同步邏輯。
- **Firebase（Firestore + Auth）**：能力對齊 Supabase，但不是 SQLite，且綁定 Google 生態。
- **繼續用 Supabase 但加 keep-alive**：用 GitHub Actions / UptimeRobot 每日 ping 一次資料庫防止 pause，
  幾乎零改動，是「最小代價先止血」的選擇（治標）。

## 5. 遷移成本評估（若走選項 1 / 2）

| 區塊 | 是否需改 | 說明 |
| --- | --- | --- |
| `modules/sync-core.js` | ✅ 不用改 | LWW 合併、build/apply snapshot 與後端無關，完整複用 |
| `modules/sync-supabase.js` | ⚠️ 重寫 | 把 `pull/pushSnapshot`、`auth`、`subscribeSnapshotChanges` 換成新後端實作（約 127 行） |
| `features/sync/sync.js` | ⚠️ 局部改 | auth UI / session 處理那段（signIn/up/OTP/reset）需對接新 auth；realtime 改輪詢或新 SDK |
| `ai-config.example.js` | ⚠️ 小改 | `SUPABASE` 換成新後端設定（如 `SYNC_ENDPOINT` / token） |
| 後端 | 🆕 新增 | D1：一個 Worker + 一張表；PocketBase：部署一個執行檔 |
| 文件 | ⚠️ 更新 | `docs/` 同步相關說明、`CLAUDE.md` 同步模組描述 |

> 重點：**最髒的合併/衝突邏輯不用動**，遷移風險主要在 auth 與 realtime 這兩塊的替換。

## 6. 建議

1. **先確認你是否真的需要跨裝置同步**（決定走第 3 節哪個場景）。
2. 若 **只單機用** → 場景①：關閉雲同步 + 用 `local-backup.js` 做手動備份。最省事。
3. 若 **要跨裝置 + 想要 SQLite** → 選項 1：**Cloudflare D1 + Worker**（推薦，根治閒置暫停、命中 SQLite 訴求）。
4. 若 **要跨裝置 + 想要一站式（含 Auth/Realtime）且願意自架一台機器** → 選項 2：**PocketBase**。
5. 若 **只想先止血、暫不改架構** → 對現有 Supabase 加每日 keep-alive ping，避免 7 天 pause。

> 「瀏覽器內本地 SQLite 取代 Supabase」這條路本身**不建議**，因為它無法承接 Supabase 的核心價值。
