# AI 提示詞跨模型 JSON 穩定性與分類重構研究

日期：2026-06-13
範圍：`modules/api.js`、`ai-config.example.js`（AI_PROMPTS / AI_TASKS）、`features/qa/qa-checker.js`、`features/dictation/dictation-grader.js`、`features/assistant/assistant.js`、`modules/validate.js`、`docs/prompts.md`

---

## 一、問題定義

1. **跨模型 JSON 穩定性差**：GPT / Grok 系列能穩定返回合格 JSON，其他模型（Gemini、GLM、Qwen、DeepSeek、各種小模型）經常返回不合格輸出（混入解釋文字、code fence、截斷、格式錯誤、把內容放錯欄位）。
2. **提示詞分類混亂**：提示詞散落多處、分類體系不一致、有兩份「真相來源」，維護成本高。

---

## 三、為什麼弱模型返回的 JSON 比較差（根因分析）

按影響程度排序，**前三項是工程問題而非提示詞問題**：

### 根因 1：`max_tokens` 太緊 → JSON 被截斷（最常見、最隱蔽）

現有限額：wordTooltip 280、sentence 精簡 380 / 完整 650、phrase 260、example 600。

- GPT 輸出緊湊，280 tokens 夠用；很多開源/小模型輸出囉嗦（喜歡加說明、欄位值偏長），或是 **reasoning 模型把思考也算進 completion tokens**（GLM-4.x、Qwen-thinking、DeepSeek-R1 類），到限額直接截斷 → JSON 永遠解析失敗。
- 程式目前**不檢查 `finish_reason === 'length'`**，截斷與「模型不會寫 JSON」混為一談，於是誤判成「這個模型不行」。

### 根因 2：`reasoning_content` 沒有統一處理

`qa-checker.js:326-331` 已經知道這件事（content 空時改讀 `reasoning_content`），但 `modules/api.js` 所有路徑都只讀 `choices[0].message.content`。GLM / Qwen / DeepSeek 走 api.js 的任務（文章分析、單詞、例句）拿到空字串 → 直接失敗。**同一個修復存在於一個檔案、缺席於另一個**，這是分裂維護的直接代價。

### 根因 3：`response_format: json_object` 無能力探測與降級（api.js 路徑）

- `qa-checker.js:286-322` 有「先帶 json_object，400/422 就重試不帶」的降級邏輯；
- `api.js` 的 analyzeParagraph / analyzeSentence / analyzeWordInSentence / analyzeSelection **一律硬帶** `responseFormat: {type:'json_object'}`，不支援該參數的端點/模型直接 400，然後被當成「模型失敗」切到 fallback 模型——如果 fallback 也不支援，整條鏈全滅。
- 部分 OpenAI 相容閘道（OpenRouter 上的某些模型、Ollama 舊版、各種轉發站）對 `response_format` 支援參差，這正好解釋「GPT/Grok 好、其他差」：GPT/Grok 官方端點完整支援 json_object，弱模型常經過相容層。

### 根因 4：頂層返回**陣列**的提示詞與 json_object 衝突

`exampleGeneration` 要求返回 `[{...}, ...]` 頂層陣列：
- `json_object` 模式規定頂層必須是 object，支援該模式的模型會被迫改寫或出錯；
- `api.js` 的 `extractJSONObject()` 只找 `{...}`，**不會擷取 `[...]`**——模型若在陣列前後加了半句說明，解析必敗。

### 根因 5：格式契約寫法對弱模型不友好

現有模板的共同寫法：中文長段落 + 「格式為: {"key":"..."}」省略號示例。

- **省略號示例**（`"phonetic":"IPA"`、`"meaning":"..."`）對 GPT 是 schema，對弱模型常被當成字面值照抄，或自由發揮鍵名（所以 normalize 層才需要猜 `translation/chinese_translation/translated_text/output/result` 五種鍵名——這是在為提示詞的含糊買單）；
- 指令與資料混在同一條 user message，大多數任務**沒有 system message**。弱模型對 system role 的服從性明顯高於混在 user 裡的要求；
- 格式要求用中文表述（「不要使用 markdown code fence」），小模型的指令遵循訓練以英文為主，英文格式契約 + 中文內容要求的組合實測更穩；
- 沒有任何 few-shot：對 7B-30B 級模型，一個**完整**的輸入→輸出示例比三段規則都有效。

### 根因 6：解析與修復層偏弱

`safeJsonParse` 只修尾逗號。弱模型常見的其他病：全形引號/冒號（`"`『』：）、字串內裸換行、單引號鍵、`True/False` 大寫、code fence 內外都有文字。qa-checker 的 extractJsonText 比 api.js 的健壯（處理 BOM、多行 fence），又是兩套各自為政。

### 根因 7：失敗後的重試策略是「換模型」而不是「修對話」

Schema 驗證失敗後直接切 fallback 模型重來，從不把錯誤回饋給模型（「你上次輸出缺少 translation 欄位，只輸出修正後的 JSON」）。一次糾錯重試的成功率遠高於換一個同樣沒被教會的模型。

### 根因 8：溫度設定

exampleGeneration 用 0.7 還要求嚴格 JSON 結構——對弱模型，結構化輸出任務溫度應 ≤0.3（創意內容靠提示詞要求多樣性，不靠溫度）。

## 四、優化方案

### 4.1 工程層（不改提示詞就能大幅改善，優先做）

1. **統一回應讀取**：在 `requestAI` 內收斂為一個 `pickContent(data)`：`message.content` → `message.reasoning_content` → `output_text`，並處理 content 為陣列的情況（搬 qa-checker 的 toText 進來）。
2. **`response_format` 能力探測 + 降級**：把 qa-checker 的「json_object 失敗就裸跑」邏輯下沉到 `requestAI`，並用一個 in-memory Map 按 `provider::model` 記住「不支援」，後續請求不再白付一次 400。行有餘力可再加最高檔 `json_schema`（支援的端點直接喂 validate.js 對應的 schema），形成 json_schema → json_object → none 三級降級。
3. **截斷偵測**：檢查 `finish_reason === 'length'`，命中就以 1.5–2 倍 maxTokens 重試一次（而不是換模型）；同時把各任務 maxTokens 基線放寬（280→500 級別），弱模型的 token 計費差額遠小於整次請求報廢。
4. **統一 JSON 擷取/修復**：合併 api.js 與 qa-checker 兩套為 `modules/json-utils.js`：支援頂層陣列、平衡括號掃描（取第一段完整 JSON 而非 indexOf/lastIndexOf）、修復尾逗號、全形標點、單引號鍵、`True/False`。
5. **糾錯重試一次**：schema 驗證失敗時，帶上原輸出與錯因再問同一個模型一次（「以下輸出不符合要求：<原因>。只輸出修正後的 JSON。」），之後才切 fallback 模型。
6. **頂層一律 object**：exampleGeneration 改要求 `{"examples":[...]}`，程式端取 `.examples`。

### 4.2 提示詞層（跨模型穩定的寫法模板）

所有結構化任務統一改成「**三段式**」：

```
system（英文，短而硬）:
You are a JSON-only API. Reply with exactly one JSON object and nothing else —
no markdown fences, no explanations, no text before or after the JSON.

user 第一段（英文格式契約，逐欄位宣告型別）:
Return a JSON object with exactly these keys:
- "translation": string, Traditional Chinese (Hong Kong usage)
- "key_points": array of 1-3 short strings (may be empty)

user 第二段（中文內容規則）:
翻譯要求：香港繁體中文用字（網上/上載/電郵/巴士/的士/單車/軟件/網絡/連結/相片），不要廣東話…

user 第三段（一個完整 few-shot，輸入短句 + 完整輸出 JSON，不用省略號）

user 第四段（資料，用明確定界符包裹）:
<input>
${sentence}
</input>
```

要點：
- 格式契約用英文、內容規則用中文（雙語分工）；
- few-shot 給**完整**示例，杜絕省略號被照抄；
- 鍵名唯一且固定（消滅 normalize 層五選一的猜鍵名）；
- 資料一律用定界符（`<input>…</input>` 或 `"""`），防止文章內容被當指令；
- 結構化任務溫度統一 0.2–0.3。

### 4.3 分類重構（解決「混亂」）

**原則：一套命名走天下——以 AI_TASKS 的任務鍵為唯一主鍵**，AI_PROMPTS、taskType、AI_LIMITS、settings.ai.models 全部對齊同一組鍵：

```
wordAnalysis            單詞分析（學習頁）
exampleGeneration       例句生成
sentenceChecking        造句檢查
qaChecking              QA 校對
articleParagraph        段落翻譯（含 fallback 子模板）
articleSentence         句子詳解（withStructure/concise/fallback 子模板）
articleWord             點詞彈窗
articlePhrase           片語解析
ocrExtract              圖片取字
ocrGrading              默寫批改（json/markdown 兩個子模板 ← 收編 dictation-grader）
articleImport           HTML 抽取/清洗（system/rulesKeepImages/rulesNoImages）
assistant               助手 system prompt（← 收編 assistant.js）
```

落地方式：
1. 新建 `modules/prompts.js` 作為**唯一預設模板來源**（含上述三段式模板 + 每任務的 schema 描述），程式碼內所有硬編後備提示詞刪除；
2. `ai-config.js` 的 `AI_PROMPTS` 降級為**使用者覆蓋層**：`getPrompt(task, variant)` 先查 AI_PROMPTS，再回 prompts.js 預設——保留可客製性，消滅雙份真相；
3. `AI_TASKS` 裡 9 個 `articleXXXTranslation(Fallback)` 舊鍵收斂為上表 4 個鍵 + 統一的 `<task>Fallback` 約定（ai-models.js 的解析鏈保留舊鍵相容一個版本）；
4. `docs/prompts.md` 重寫：不再貼提示詞全文與行號，只記分類表 + 指向 `modules/prompts.js`，避免再次過期。

### 4.4 建議實施順序

| 階段 | 內容 | 預期效果 |
|---|---|---|
| P1（工程修復） | 4.1 的 1–4（reasoning_content、response_format 降級、截斷重試、統一解析） | 不改提示詞即可消除大部分「弱模型失敗」，風險最低 |
| P2（提示詞改寫） | 4.2 三段式改寫全部 JSON 任務 + 溫度/陣列頂層修正 | JSON 合格率進一步提升，normalize 猜鍵名邏輯可簡化 |
| P3（分類重構） | 4.3 prompts.js + 任務鍵收斂 + 文件重寫 | 維護成本下降，新增任務只動一處 |

P1/P2 改動集中在 `modules/api.js`、`modules/validate.js`（或新增 json-utils）與模板文本，不動各 feature 的 UI 邏輯；P3 涉及 ai-models.js 鍵名相容，建議單獨一次提交並手動巡檢文章詳解、QA 校對、OCR 三條主鏈。

## 五、驗證方式（本專案無自動化測試）

1. 在全域設定把任務模型分別切到一個強模型（gpt-4.1-mini）與一兩個弱模型（如 openrouter 上的小模型 / GLM / Qwen）；
2. 巡檢：文章詳解（段/句/詞/片語四級）、學習頁例句生成與造句檢查、QA 校對單題與整批、OCR 默寫批改 JSON 模式；
3. 觀察 console：`JSON 解析失敗` 警告應顯著減少；針對 reasoning 模型確認不再出現空回應；
4. 截斷場景：故意把 maxTokens 調低驗證 `finish_reason=length` 重試路徑。
