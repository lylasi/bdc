# 降低 `/version` 輪詢請求頻率 — 自動同步策略研究

- 時間：2026-06-23 19:28
- 範圍：登入後對 `https://bdc-sync-api.penzibdc.workers.dev/version` 的高頻輪詢
- 結論先行：**保留事件驅動的 push，改造 pull 端為「可見性感知 + 自適應退避」**，預期穩態請求量可降約 **85–95%**，零協議改動、風險低。若要徹底去除輪詢，再評估第二階段的伺服器推送（SSE / WebSocket）。

---

## 1. 現況與問題量化

### 1.1 輪詢機制
- 入口：`modules/sync-cloudflare.js` 的 `subscribeSnapshotChanges()`（約 231–260 行）。
- 間隔常數：`POLL_MS = 45000`（`sync-cloudflare.js:13`），以**遞迴 `setTimeout`** 串接，實際間隔 = 45s + 回應耗時。
- 啟動條件：僅登入後。`attachRealtime(user)` 在無 `user.id` 時直接 return（`features/sync/sync.js:1441`），於登入狀態變化／初始 session 時建立，且會先 unsubscribe 舊的再訂閱，不疊加。
- 每輪邏輯：`GET /version` → 比對本地 `lastSnapshotVersion`，遠端較新才觸發 `scheduleAutoSync('realtime')`（再被 5s 節流，`sync.js:1444-1448`）。

### 1.2 兩個關鍵缺陷
1. **背景分頁不暫停**：`main.js` 的 `visibilitychange` 只處理默寫暫停，輪詢不受影響（瀏覽器雖會節流背景 timer，但不可靠且不一致）。
2. **閒置不退避**：只要登入就固定 45s 一發，無論使用者是否在操作、資料是否在變動，永不放慢。

### 1.3 單次請求的真實成本
Worker 端每個 `/version` 請求會跑兩次 D1 讀取：
- `authenticate()`：`SELECT user_id, label FROM users WHERE token_hash = ?`（`cloudflare-worker/src/worker.js:49-52`）
- 版本查詢：`SELECT version FROM snapshots WHERE user_id = ?`（`worker.js:119`）

即 **每次輪詢 = 2 次 D1 讀**。

### 1.4 量級估算（單一登入裝置）
| 指標 | 數值 |
|---|---|
| 請求頻率 | 45s/次 → ~80 次/小時 |
| 每日請求 | ~1,920 次/裝置 |
| 每日 D1 讀取 | ~3,840 次/裝置 |
| 有效命中（真正有遠端更新）| 極低，大多數是「版本未變」的空轉 |

多裝置、多分頁時線性放大。這正是你觀察到的「不斷請求」。

### 1.5 已經做得好的部分（push 端，不需動）
- 本地資料變更經 `modules/sync-signals.js` 的 `bdc:data-changed` 事件 → `scheduleAutoSync()`。
- 已有去抖與下限：`DEBOUNCE_MS = 6000`、`MIN_INTERVAL_MS = 20000`（`sync.js:1309-1310`），push 端是事件驅動且合理，本研究**不更動**。
- 每次自動同步另存一份本機備份（localStorage，非網路；受 `MAX_AUTO_BACKUPS = 2` 約束），與請求頻率無關。

> 換言之：問題集中在 **pull 端的週期輪詢**，不是 push。

## 2. 候選方案與取捨

### 方案 0：直接調大固定間隔（45s → 180s）
- 改動：1 行常數。
- 效果：請求量降 ~4×。
- 缺點：仍背景空轉、仍不退避；只是把問題等比例縮小，沒解決結構。
- 定位：**臨時止血**，不建議作為終態。

### 方案 1：可見性感知 + 焦點即時拉取（高 CP 值）⭐
- 規則：
  - `document.hidden` → 暫停輪詢計時器。
  - `visibilitychange` 回到可見 / `focus` → **立即拉一次** `/version`，再恢復計時。
- 效果：背景分頁零請求；回到頁面立刻最新。多分頁時只有前景分頁在輪詢。
- 風險：極低（純前端、不動協議）。
- 預期：日常使用前景占比若 ~20%，請求量直接降 ~80%，且「切回即同步」體感更好。

### 方案 2：自適應退避（idle backoff）⭐
- 規則：可見且連續 N 輪「版本未變」→ 逐步拉長間隔，例如 `90s → 180s → 300s(封頂)`；任一條件命中即重置回 base：
  - 偵測到遠端版本變化；
  - 本地剛發生資料變更（`lastLocalChangeAt` 更新）；
  - 剛回到前景。
- 效果：活躍時靈敏、發呆時安靜。
- 風險：低；需小心重置條件，避免「永遠停在最大間隔」。

### 方案 3：純事件驅動，移除週期輪詢
- 規則：只在「回到前景 / 視窗 focus / 本地 push 後」各拉一次，平時完全不輪詢。
- 效果：請求量最低。
- 缺點：兩台裝置同時開著、都不切換分頁時，跨裝置更新不會即時可見（要切走再切回才更新）。對本專案「個人多裝置、非協作即時」場景多半可接受，但屬行為改變，需你確認。

### 方案 4：伺服器推送（SSE 或 WebSocket / Durable Objects）
- 作法：Worker 端開 `text/event-stream`（SSE）或用 Durable Object 維持 WebSocket，版本變更時主動推。
- 效果：徹底消滅輪詢，真正即時。
- 缺點：
  - Worker 需改造（SSE 長連線 / DO 計費與連線數）、CORS 與斷線重連處理。
  - 長連線本身也有成本與複雜度，且要處理睡眠分頁的重連。
- 定位：**第二階段**，只有在「即時跨裝置」成為硬需求時才值得。

### 方案 5（正交優化）：讓單次請求更便宜
- `/version` 已很輕，但每次仍含 1 次 token 認證 D1 讀。可考慮：
  - 回應加 `ETag` / 支援 `If-None-Match` → 未變回 `304`（仍有一次往返，但語義更清晰、可被中介快取）。
  - 以短 TTL 在 Worker（KV / 記憶體）快取 token→user 映射，省掉認證那一次 D1 讀。
- 效果：把每次成本從 2 D1 讀降到 ~1。與方案 1/2 正交，可疊加。
- 定位：錦上添花，非當務之急。

## 3. 推薦組合（建議落地）

**第一階段：方案 1 + 方案 2（可選疊加方案 5）**

行為定義：
1. 背景（`document.hidden`）→ 停止輪詢。
2. 回到前景 / `focus` → 立即拉一次，並把間隔重置回 base。
3. 前景閒置時自適應退避：`base → 2×base → 4×base`，封頂 `maxIntervalMs`。
4. 偵測到遠端版本變化或本地資料變更 → 間隔重置回 base。
5. 全部參數可由 `ai-config.js` 覆寫，給預設值。

建議預設：
```
pollBaseMs = 90000      // 90s（取代固定 45s）
pollMaxMs  = 300000     // 5min 封頂
pollOnFocus = true      // 回前景立即拉一次
pauseWhenHidden = true  // 背景暫停
```

預期：穩態請求量較現況降 **~85–95%**（背景歸零 + 前景退避），且「切回前景即最新」反而更跟手。

**第二階段（按需）**：若日後需要真正的即時跨裝置同步，再評估方案 4（SSE 優先於 WebSocket，改動較小）。

## 4. 落地改動點（供實作參考）

| 位置 | 改動 |
|---|---|
| `modules/sync-cloudflare.js` `subscribeSnapshotChanges()` | 核心改造：間隔改為可變、加入 `visibilitychange` / `focus` 監聽、退避與重置邏輯；`POLL_MS` 改讀設定。 |
| `modules/settings.js` / `ai-config.example.js` | 新增 `sync.pollBaseMs / pollMaxMs / pollOnFocus / pauseWhenHidden` 設定與預設值與註解。 |
| `features/sync/sync.js` `attachRealtime()` | 維持現有訂閱／反訂閱生命週期；確保登出時正確停掉新加的監聽器（避免 listener 洩漏）。 |
| （可選）`cloudflare-worker/src/worker.js` | 方案 5：`/version` 加 `ETag`/`304`，或快取 token 認證。 |

注意事項：
- 監聽器需在 `unsubscribeChannel`/`stop()` 時一併移除，避免重複 `attachRealtime` 造成累積。
- 退避重置條件要涵蓋「本地剛改資料」，否則使用者剛編輯完卻要等到最大間隔才上行偵測。
- 多分頁：方案 1 已天然讓背景分頁噤聲；如需更嚴格的單一主導分頁，可用 `BroadcastChannel` 選主（非必要，列為後續）。

## 5. 驗證方式（無自動化測試，手動巡檢）
1. 登入後開 DevTools Network，過濾 `/version`：
   - 切到其他分頁/最小化 → 應**停止**新請求；切回 → **立即**一次。
   - 前景發呆數分鐘 → 間隔應逐步拉長到封頂。
   - 在另一裝置改資料 → 本裝置回前景或下一輪應偵測到並同步。
2. 編輯本地資料 → push 端去抖（6s）行為不變、間隔重置回 base。
3. 登出 → 確認輪詢與所有新監聽器都停止（無殘留請求）。
4. 回歸：對照 `CLAUDE.md` 同步相關回歸區（登入/同步/本地備援）。
