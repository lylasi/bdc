# 自动同步触发操作清单

## 结论摘要

当前自动同步的触发源不是“定时轮询”，而是**事件驱动**：本地数据变更会发出 `bdc:data-changed`，同步模块监听后调用 `scheduleAutoSync()`。

关键入口：
- `features/sync/sync.js:47-52`
- `modules/sync-signals.js:3-10`

自动同步调度参数：
- 防抖：`6s`（`features/sync/sync.js:1403`）
- 最小间隔：`20s`（`features/sync/sync.js:1404`）
- 远端 realtime 事件节流：`5s`（`features/sync/sync.js:1488-1492`）

## 一、会触发自动同步的用户操作

### 1. 词书相关

这些操作最终会调用 `saveVocabularyBooks()` 或 `saveAppState()`，从而触发：
- `syncTouch('vocabulary')`
- 入口：`modules/storage.js:99-111`、`modules/storage.js:386-398`

具体用户动作：

1. **切换当前词书**
   - `features/vocabulary/vocabulary.js:175-185`
   - 点击左侧词书列表切换选中项，会 `saveAppState()`。

2. **导入词书 / 批量导入词书**
   - `features/vocabulary/vocabulary.js:645-649`
   - 导入成功后 `saveVocabularyBooks()`。

3. **新建或编辑词书并保存**
   - `features/vocabulary/vocabulary.js:722-788`
   - 保存后 `saveVocabularyBooks()`。

4. **删除当前词书**
   - `features/vocabulary/vocabulary.js:822-836`
   - 会同时 `saveVocabularyBooks()` 和 `saveAppState()`。

5. **补完缺失词条**
   - `features/vocabulary/vocabulary.js:302-343`
   - 批量补词后 `saveVocabularyBooks()`。

6. **词书内去重合并**
   - `features/vocabulary/vocabulary.js:345-392`
   - 去重完成后 `saveVocabularyBooks()`。

7. **其他增删改单词操作**
   - 例如 `modules/storage.js:202-279`
   - `saveWordbook()` -> `saveVocabularyBooks()`。

8. **通过学习页把单词写回词书**
   - `features/learning/learning.js:124`
   - 会触发 `storage.saveVocabularyBooks()`。

9. **通过 vocab 模块添加默认词书 / 当前文章词书**
   - `modules/vocab.js:56-57`
   - `modules/vocab.js:206`
   - 也会触发词书同步。

### 2. 文章分析相关

这些操作会触发 `syncTouch('articles')`：
- `modules/storage.js:28-35`
- `modules/storage.js:420-431`
- `modules/storage.js:508-532`

具体用户动作：

1. **保存文章 metadata**
   - 如文章来源、标题、shareId 之类
   - `modules/storage.js:58-72`
   - 注意：这会触发同步，但 `articleMetas` **本身并未进入远端 snapshot**。

2. **删除文章 metadata**
   - `modules/storage.js:75-81`
   - 同样会触发同步，但 metadata 本身不上云。

3. **保存文章分析结果**
   - `modules/storage.js:508-532`
   - 这部分会进入远端 `articles.analyzedArticles`。

4. **删除单篇分析记录**
   - `modules/storage.js:442-454`
   - 内部调用 `saveAnalyzedArticles()`。

5. **清空文章分析记录**
   - `modules/storage.js:456-465`
   - 也会触发 `articles` 分组同步。

### 3. QA 问答集相关

这些操作会触发 `syncTouch('qa')`：
- `features/qa/qa-storage.js:131-136`
- `features/qa/qa-storage.js:176-190`
- `features/qa/qa-storage.js:201-230`

具体用户动作：

1. **保存 QA 集**
   - 新建 / 编辑后保存
   - `features/qa/qa-storage.js:176-198`

2. **导入 QA 集**
   - `features/qa/qa-storage.js:241-263`
   - 导入成功后调用 `saveQASet()`。

3. **删除 QA 集**
   - `features/qa/qa-storage.js:201-238`
   - 在 UI 上对应 `features/qa/qa.js:1630`。

4. **更新 QA manifest**
   - `features/qa/qa-storage.js:340-369`
   - 保存/删除时会顺带改 manifest 并再次触发一次 `saveQASetsManifest()`。

### 4. AI assistant 会话相关

assistant 目前通过 `writeIndex()` 触发：
- `features/assistant/assistant.js:213-214`
- 这里会 `syncTouch('assistant')`

具体用户动作：

1. **创建新会话**
   - `features/assistant/assistant.js:703-710`
   - `ensureMetaWithId()` -> `writeIndex()`。

2. **重命名会话**
   - `features/assistant/assistant.js:762-769`
   - `writeIndex()`。

3. **删除会话**
   - `features/assistant/assistant.js:772-779`
   - `writeIndex()`。

4. **旧版 assistant 数据迁移**
   - `features/assistant/assistant.js:441-457`
   - 启动时如果发现旧键，会迁移并 `writeIndex()`。

注意：
- 单纯 `idbSetConv()` 写消息正文本身**不会发 syncTouch**。
- 但发送消息时通常会更新索引时间戳，因此仍可能通过 `writeIndex()` 间接触发。
- 更重要的是：当前远端 snapshot 仍读旧键 `assistantConversations`，与现行 `assistantConvIndex + IndexedDB` 存储不一致，导致 assistant 自动同步实现疑似不完整。
  - `modules/sync-core.js:101-115`
  - `features/assistant/assistant.js:15-18`

### 5. 听写设置相关

这些操作会触发 `syncTouch('dictation')`：
- `modules/state.js:272-285`

具体用户动作：

1. **切换普通话 / 粤语朗读选项**
   - `features/dictation/dictation.js:76-99`
   - 变更时会 `saveDictationSettings()`。

2. **开始听写前保存当前设置**
   - `features/dictation/dictation.js:197-206`
   - 即开始听写时也会触发一次。

3. **其他听写设置保存点**
   - `features/dictation/dictation.js:479`
   - 以及 `state.saveDictationSettings()` 的其他调用点。

注意：
- 当前 snapshot 里进入远端的是 **dictation settings**，不是完整听写 session。
- 也就是会同步 `repeatTimes / wordInterval / voice / loopMode / shuffleMode / listenOnlyMode / showWordInfo / updatedAt` 等设置。

### 6. 听写批改历史相关

这些操作也会触发 `syncTouch('dictation')`：
- `modules/storage.js:477-499`

具体用户动作：

1. **新增一条批改历史**
   - `modules/storage.js:477-486`
   - UI 入口见 `features/dictation/dictation-grader.js:247`

2. **删除一条批改历史**
   - `modules/storage.js:489-494`
   - UI 入口见 `features/dictation/dictation-grader.js:76`

3. **清空全部批改历史**
   - `modules/storage.js:496-499`
   - UI 入口见 `features/dictation/dictation-grader.js:45`

但注意：
- 这些历史记录**不会进入远端 snapshot**。
- 它们只是误触发了 `dictation` 分组的自动同步。

## 二、不依赖本地改动、也会自动触发同步的情况

### 1. 登录成功后
- `features/sync/sync.js:70-79`
- 登录后会 `scheduleAutoSync('login')`

### 2. 初始化发现已有登录 session
- `features/sync/sync.js:86-97`
- 会 `scheduleAutoSync('init-session')`

### 3. 远端 snapshots 表发生 realtime 变化
- `features/sync/sync.js:1483-1493`
- 会 `scheduleAutoSync('realtime')`
- 前面有一层 5 秒 throttle，但仍可能在多人/多端场景下较频繁触发。

## 三、不会触发自动同步的常见操作

1. **只读操作**
   - 浏览词书、浏览文章、切页但不改数据。

2. **导出操作**
   - 导出词书：`features/vocabulary/vocabulary.js:879-914`
   - 导出 QA：`features/qa/qa-storage.js:270-305`
   - 这些只是下载，不写本地状态。

3. **assistant 仅切换查看会话**
   - 打开历史、切换到某个既有会话，本身不一定触发 `writeIndex()`。

4. **听写 session 的保存/恢复**
   - `saveDictationSession()` / `loadDictationSession()` 主要是本地 session，不直接触发 `syncTouch()`。

## 四、我认为最值得注意的“高频误触发点”

### 1. articleMetas
- 会触发 `articles` 自动同步
- 但并未进入远端 snapshot
- 属于“触发了但没必要”

### 2. dictationGradingHistory
- 会触发 `dictation` 自动同步
- 但并未进入远端 snapshot
- 也是“触发了但没必要”

### 3. QA 保存可能连触发两次
- `saveQASet()` 自己触发一次 `syncTouch('qa')`
- 它内部 `updateQASetsManifest()` -> `saveQASetsManifest()` 又会再触发一次
- 虽然有 debounce，但仍然增加调度噪音

### 4. assistant 当前实现前后不一致
- 触发了 `assistant` 分组同步
- 但 snapshot 读取的键和当前真实存储结构不一致
- 可能造成“频繁触发，但同步不到真实会话内容”

## 五、最终结论

从“哪些操作会触发自动同步”来看，当前系统属于：

- **触发面很广**：词书、文章、QA、assistant、听写设置、听写批改历史都会触发
- **其中有一部分是有效同步**：词书、文章分析、QA、听写设置
- **也有一部分是无效/低价值触发**：articleMetas、dictationGradingHistory、以及疑似不完整的 assistant 同步

如果后续要优化，优先级建议是：
1. 去掉不会上云的数据变更触发（articleMetas、grading history）
2. 修正 assistant snapshot 读取来源
3. 让 QA 保存合并成一次触发
4. 再调整同步节流参数
