# 文章詳解頁三級互動（點詞/句子/段落）問題調研

日期：2026-06-13
前置：本報告聚焦文章詳解頁，與《20260613022852478_AI提示詞跨模型JSON穩定性與分類重構研究.md》互補——那份講「AI 返回的 JSON 不穩」，這份講「就算 AI 返回正確，前端邏輯也會把結果用錯」。

核心結論：**文章詳解頁「經常有問題」的主因不在提示詞，而在前端有 4 個正確性 bug / 設計缺陷**，其中兩個直接命中「點擊單詞結合句子的解釋不準」這個痛點。

---

## 一、點詞解釋（單詞 + 句子）——兩個正確性 bug

### Bug 1：命中查找只按單詞、不按句子（最關鍵）

`features/article/article.js:2853-2854`：

```js
const matchingAnalyses = analysisArray.filter(item => item.word && item.word.toLowerCase() === wordText.toLowerCase());
let wordAnalysisData = (wordIndex < matchingAnalyses.length) ? matchingAnalyses[wordIndex] : null;
```

- 查找 `dataset.analysis` 時只比對 word，**完全不比對句子**；
- `wordIndex` 來自 `data-word-index`，但渲染函數（`wrapWordsInTextHTML`、`wrapWordsInElementOnce`）**從未設置過這個屬性**，恆為 0；
- 結果：同一單詞第二次在**任何其他句子**被點擊，永遠直接返回**第一次點擊那句**的解釋（詞性、意思、語法作用全是舊句子的），根本不會發 AI 請求。

這正是「點擊單詞結合句子的解釋，以明確單詞的解釋」失效的直接原因——多義詞（run/charge/present…）在第二個句境裡顯示的是第一個句境的釋義，且看起來像 AI 答錯了，實際是前端拿錯了快取。

### Bug 2：句子定位用樸素正則 + 子字串匹配，與渲染分句不一致

`article.js:2894-2895、2910-2911`：

```js
const sentences = paraEnglish.match(/[^.!?]+[.!?]*/g) || [paraEnglish];
const sentence = sentences.find(s => s.includes(wordText)) || paraEnglish;
```

三個問題疊加：

1. **`includes` 是子字串匹配不是整詞匹配**：點 "he" 會命中第一個含 "the/she/here" 的句子；點 "art" 會命中含 "start" 的句子 → 送給 AI 的「句子」根本不是使用者點的那句，AI 給出的句境解釋自然「不準」；
2. **縮寫切錯**：`Mr. / U.S. / e.g.` 會被 `[.!?]` 切斷；
3. **與渲染用的分句器不一致**：頁面渲染句子 span 用的是 `sentenceSplit()`（Intl.Segmenter，article.js:3397），點詞時卻另起爐灶用樸素正則重新分句。兩套分句結果不同 → 點詞用的 sentence 字串與句卡、IndexedDB 快取鍵全部對不上，**同一句的詞重複請求、快取永不命中**。

諷刺的是正確做法已經存在於同檔案：`ensureWordPairMapping()`（article.js:2974-2978）就是直接取 `wordEl.closest('.interactive-sentence').dataset.sentence`——點詞元素本來就在帶 `data-sentence` 屬性的 span 裡，根本不需要重新分句。

### 缺陷 3：滑鼠 hover 就觸發完整句子 AI 分析

`handleWordHighlight`（article.js:2598-2620）：mouseover 一個無 pairId 的詞 → `ensureWordPairMapping` → **發起一次完整句子分析請求**（650 maxTokens，只為找中文對照詞）。

- 滑鼠掃過一個段落 = 每句一個 AI 請求（有按句去重，但句句都會發）；
- `AI_LIMITS` 預設併發只有 2：hover 觸發的請求**擠佔同一隊列**，使用者真正點擊的詞/句卡排在後面 →「載入中...」轉很久甚至超時，這是「經常有問題」的另一個直接來源；
- hover 請求結果只用了 `phrase_alignment` 一個欄位，性價比極低。

### 缺陷 4：失敗無重試、靜默降級

點詞失敗只顯示「分析數據未找到」（2883），無重試按鈕（句卡有，tooltip 沒有）。

## 二、句子分析

1. **冗餘請求**：點一個詞最多同時跑兩條 AI 鏈（hover 的句子分析 + click 的單詞分析），而句子分析結果裡其實常已含該詞資訊，互不復用。
2. **中英高亮按 index 1:1 硬對齊**：英文用 `sentenceSplit()`、中文用 `sentenceSplitZh()` 各自分句後按序號配對（1846-1873）。AI 翻譯併句/拆句（很常見：英文兩短句譯成一個中文長句）後 index 錯位 → 點英文第 3 句高亮中文第 3 句但內容對不上，視覺上就是「混亂」。
3. **依賴弱模型最不穩的欄位**：`phrase_alignment` / `chunks` 正是弱模型最常缺漏的結構（上一份報告根因 5），缺了之後 hover 配對、句卡分塊就時靈時不靈。

## 三、段落翻譯

1. **`\b` 模板字串 bug（死代碼）**：`article.js:1854、1860`

   ```js
   new RegExp(`\b(${escapeRegex(pair.en)})\b`, 'g')
   ```

   模板字串中 `\b` 是**退格字元 U+0008**，不是正則的單詞邊界（需 `\\b`）。這兩個正則永遠匹配不到任何東西 → 基於 `word_alignment` 的中英詞配對包裝**從未生效過**。
2. **與 prompt 已脫節**：段落分析現在用 `level: 'quick'`，prompt 明確要求「不要返回 word_alignment 與 detailed_analysis」，所以上面那段對齊代碼即使沒有 `\b` bug 也無數據可用——bug 與死代碼互相掩護，應一併清理。
3. **失敗文案污染持久化數據**：段落失敗時把 `[第 N 段分析失敗]` 寫進 `chinese_translation` 並隨 `saveAnalysisResult` 存入歷史與同步快照；之後從歷史載入，失敗文案就成了「譯文」。
4. 併發 2 + 400ms 間隔 + 失敗段可重試，這部分設計是合理的，不需要動。

## 四、優化方案

### P0：修正確性（小改動、高收益，全部在 article.js）

| # | 修改 | 位置 |
|---|---|---|
| 1 | 點詞 sentence 一律取 `closest('.interactive-sentence').dataset.sentence`，刪除兩處樸素正則分句 | 2894-2895、2910-2911 |
| 2 | 命中查找改為 **word + sentence 雙鍵**匹配（`analysisArray.find(it => it.word===w && it.sentence===s)`），找不到才懶載；`_lazyCache` 鍵同步 | 2853-2854、2914 |
| 3 | hover 不再觸發 AI：`ensureWordPairMapping` 只在已有句子快取（cache 命中）時做配對，否則僅本地高亮；或乾脆把配對改到句卡展開後進行 | 2598-2620、2970 |
| 4 | 修 `\b` → 連同已無數據來源的 word_alignment 包裝邏輯一起刪除 | 1850-1862 |
| 5 | 段落失敗不把 `[第 N 段分析失敗]` 寫進持久化結果（渲染層顯示失敗狀態即可，保存時留空） | 1687、1703-1715 |
| 6 | tooltip 失敗加「重試」按鈕（句卡已有同款） | 2883、2961-2964 |

### P1：結構優化——「按句翻譯」一次解決段落+句子兩級

把段落 prompt 從「整段譯文一個字串」改為**逐句輸出**：

```json
{"sentences": [{"en": "原句", "zh": "譯句"}, ...]}
```

收益：
- 中英句高亮**精確對齊**（不再靠兩套分句器 index 硬配），徹底解決錯位混亂；
- 句卡展開可先即時顯示該句譯文（已有數據），再懶載 chunks/key_points，體感快很多；
- 段落譯文 = `sentences.map(s=>s.zh).join('')`，原渲染不受影響；
- 弱模型輸出「en 抄原文 + zh 譯文」的成對結構，遠比輸出 `phrase_alignment` 可靠。

配套：點詞時優先從已有的句級數據（句卡快取/persistSentenceAnalysis 結果）復用資訊，沒有才發單詞請求——把三級從「三條獨立 AI 鏈」變成「逐級復用的一條鏈」。

### P2：與上一份報告的工程修復銜接

文章詳解的三個任務（articleParagraph/Sentence/Word）正是上一份報告 P1 工程修復（reasoning_content 統一讀取、response_format 降級、截斷重試、maxTokens 放寬：word 280→500、sentence 650→900）+ P2 三段式提示詞改寫的最大受益者，建議按 P0（本報告）→ P1 工程（上報告）→ 按句翻譯（本報告 P1）順序實施。

## 五、驗證清單（手動巡檢）

1. 同一個多義詞出現在兩個句子，分別點擊 → 兩次都應發請求（或命中各自句子的快取），解釋隨句境不同；
2. 點擊 "he"、"art" 等易誤配的短詞 → tooltip 的「片語解析」按鈕 data-sentence 應為所點的那句；
3. 滑鼠快速掃過整段 → Network 面板不應出現成串句子分析請求；
4. 含 Mr. / U.S. 的句子 → 句卡與點詞使用的句子邊界一致；
5. 段落分析中途斷網 → 失敗段顯示重試，從歷史重新載入時譯文不含「[第 N 段分析失敗]」；
6. （若實施按句翻譯）英文兩短句譯成一中文長句的段落 → 高亮對應正確。
