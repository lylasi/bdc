# BDC 專案改進分析

> 日期：2026-06-13 · 範圍：全倉庫靜態巡檢（架構、安全、資料層、倉庫衛生）

## 一、安全（優先處理）

### 1. `modules/markdown.js` 存在 XSS 漏洞 — 高
- `esc()`（modules/markdown.js:6）只轉義 `& < >`，**不轉義引號 `"`**。
- 連結語法 `[text](url)` 的 href 直接插入 `$2`（modules/markdown.js:67），導致兩個攻擊面：
  1. 屬性逃逸：`[a](" onmouseover=alert(1) x=")` → `href=""onmouseover=...`
  2. 危險 scheme：`[a](javascript:alert(1))` 未過濾。
- 此函式渲染的是 **AI 回傳內容與 OCR 結果**（assistant、OCR 預覽），屬不可信輸入，是真實可達的攻擊路徑。
- 修法：`esc` 加入 `"` 與 `'` 的轉義；href/src 白名單 scheme（http/https/data:image、相對路徑）。圖片 `src`（markdown.js:14）同樣需要 scheme 檢查。

### 2. 各功能模組自行手寫 escapeHtml，實作不一致 — 中
- 至少四份獨立實作：`features/qa/qa-creator.js:252`、`features/assistant/assistant.js:950`、`features/article/article.js:1458`、`features/sync/sync.js:1184`。
- 全倉庫 `innerHTML` 使用約 197 處（article.js 63 處、vocabulary.js 27 處、sync.js 22 處、qa.js 22 處…），是否每一處都正確套用轉義難以稽核。
- 修法：把 `escapeHtml` 收斂到 `modules/`（如 `modules/ui.js` 或新 `modules/html.js`）統一匯出，逐步替換。

### 3. `features/assistant/assistant.js:550` 直接 fetch AI 端點 — 中
- 違反「所有 AI 請求走 `modules/api.js`」的專案邊界（CLAUDE.md 明文規範），會繞過統一的 key 管理、錯誤處理與快取。

## 二、資料層穩健性

### 4. `JSON.parse` 無防護，壞資料會炸掉啟動流程 — 高
- `modules/storage.js:169`（vocabularyBooks）、`storage.js:473`（analyzedArticles）等多處直接 `JSON.parse(saved)` 無 try/catch。
- 這些函式由 `main.js` 啟動時呼叫；任一 localStorage key 損壞（手動編輯、同步寫入中斷、quota 截斷）會讓整個 SPA 白屏。
- 修法：包一層 `safeParse(key, fallback)`，失敗時保留原始值到 `<key>.corrupt` 並回退預設值。

### 5. 完全沒有 localStorage 容量（QuotaExceededError）處理 — 中
- 全倉庫 grep `quota` 零命中。文章分析結果、QA set、AI 快取回退時都會寫 localStorage，超量時 `setItem` 直接 throw，且使用者無感知資料未存入。
- 修法：集中 `modules/storage.js` 的寫入路徑，捕捉 QuotaExceeded 並提示使用者（並考慮把大物件遷往 IndexedDB，cache.js 已有先例）。

### 6. 同步合併為 updatedAt 字串比較的 last-write-wins — 中（設計取捨，需知情）
- `modules/sync-core.js` 以 `String(updatedAt).localeCompare` 決定去留（sync-core.js:152-170、214-215）。
- 多裝置同時編輯同一詞書/QA set 時，較舊裝置的整組變更會被整體覆蓋（snapshot 粒度，非欄位級合併）。對單人多裝置場景可接受，但建議：
  - 在文件中明示此語義；
  - 覆蓋前自動經 `modules/local-backup.js` 留一份本地快照，給使用者反悔的機會。

## 三、架構與可維護性

### 7. 巨型檔案，職責過載 — 中
| 檔案 | 行數 | 問題 |
|---|---|---|
| `features/article/article.js` | 3840 | 載入、分析、渲染、朗讀、互動句子全在一檔；114 處 DOM 查詢、63 處 innerHTML |
| `features/qa/qa.js` | 2565 | 已有 qa-storage/qa-trainer/qa-checker/qa-pdf 拆分先例，但主檔仍過大 |
| `features/sync/sync.js` | 1653 | UI 與同步流程混雜 |
| `modules/api.js` | 1410 | 所有 AI 端點 + 各功能 prompt 集中於一檔 |
| `styles.css` | 6406 | 全部模組樣式單檔 |
| `index.html` | 1447 | 九個模組 DOM 全部預載 |

- 建議優先拆 `article.js`：渲染（escapeHtml/表格/互動句）、分析流程、朗讀控制三塊邊界清楚。
- `styles.css` 可按 feature 拆成 `features/<m>/<m>.css` 再以多個 `<link>` 引入（無 bundler 也可行）。

### 8. DOM 查詢散落，違反自身慣例 — 低
- CLAUDE.md 規定 DOM 參照集中 `modules/dom.js`（僅 175 行），但 features 下有 480+ 處 `querySelector/getElementById`（article 114、sync 105、qa 102）。規範已名存實亡，建議二選一：認真收斂，或修訂 CLAUDE.md 改為「模組內自查、跨模組才進 dom.js」。

### 9. 吞錯式 catch 普遍 — 低
- `catch(_){}` 模式大量存在（assistant.js 尤甚），article.js 144 個 catch。出錯時使用者與開發者都無感。建議至少統一走一個 `reportError(err, context)`（debug 模式 console、必要時 toast）。

## 四、倉庫衛生

### 10. 開發殘留物已提交 — 低（易修）
- 應移除並加入 .gitignore：`console-after-settings.txt`、`console-final.txt`、`playwright-console-sync.txt`、`playwright-global-settings.md`、`settings-snapshot.md`、`global-settings-*.png`、根目錄 `5AU1_1.jpg`/`5AU1_2.jpg`。
- `package-lock.json` 被追蹤但**沒有 package.json**，純屬殘留。
- `.gitignore` 列了 `CLAUDE.md` 但該檔已被追蹤（且專案明確希望它入庫）——應把這行從 .gitignore 移除，避免誤導。

### 11. 本機 `ai-config.js` 含多組真實 API key — 提醒
- 該檔未被 git 追蹤（正確），但檔內留有多組已註解的歷史 key。建議清理失效 key；若任何 key 曾被 commit 過歷史版本，應視為已洩漏並輪替。

## 建議處理順序

1. **馬上**：修 markdown.js XSS（#1）＋ storage.js safeParse（#4）——兩者都是小改動、高風險消除。
2. **本週**：倉庫衛生清理（#10）、escapeHtml 收斂（#2）、assistant fetch 收編進 api.js（#3）。
3. **漸進**：Quota 處理（#5）、同步覆蓋前自動備份（#6）、article.js 拆分（#7）。
4. **擇期決策**：dom.js 慣例去留（#8）、錯誤回報統一（#9）、styles.css 拆分（#7）。
