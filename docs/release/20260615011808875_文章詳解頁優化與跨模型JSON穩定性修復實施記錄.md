## 背景

文章詳解頁（點詞解釋、句子分析、段落翻譯）長期「經常有問題」，且「GPT/Grok 正常、其他模型很差」。調研定位到兩類根因：

1. **前端正確性 bug**：點詞拿錯句子、快取拿錯結果、hover 觸發大量 AI 請求、中英高亮錯位。
2. **跨模型 JSON 不穩**：主因不在提示詞，而在工程層——reasoning 模型（gpt-oss / GLM / Qwen / DeepSeek）把「思考」算進 completion 額度，導致 JSON 被 `max_tokens` 砍斷（`finish_reason==='length'`），而程式把截斷殘骸當答案顯示並快取。

本次按 P0（正確性）→ P1（按句翻譯）→ 工程修復 三批落地。

## P1：段落「按句翻譯」對齊（`api.js` + `article.js` + `validate.js` + `ai-config.example.js`）

**目標**：徹底解決中英句高亮錯位——舊版英文用 `sentenceSplit()`、中文用 `sentenceSplitZh()` 各自分句後按序號硬配對，模型併句/拆句即錯位。

**設計**（比「要求模型回显英文」更穩）：前端先用與渲染同款分句器切好句子、編號後交給模型，模型只返回**等長的中文譯句陣列**，對齊由構造保證、省 token、避免回显不一致。

- `api.js / analyzeParagraph`：新增 `opts.sentences`（按句模式）。prompt 改為「編號句子 → `{"sentences":[...]}` 等長陣列」；`normalizeSentenceTranslations()` 嚴格校驗「陣列等長且全非空」，不合格自動降級到原整段翻譯鏈。返回值附 `sentences:[{en,zh}]`，`chinese_translation` 由 zh 派生。快取鍵加 `-sent` 後綴，不與舊快取互污。
- `article.js`：
  - 只對**純文字段落**啟用（`isPlainProseParagraph()` 排除表格/清單/標題/引用/圖片，保護 Markdown 結構）。
  - `alignedZhSentences()`：分析結果帶 `sentences` 且與本地分句逐句吻合（長度 + en 比對）時，中文側按其渲染；否則回退 `sentenceSplitZh()`。`renderParagraph` 與 `displayArticleAnalysis` 兩條渲染路徑都接入。
  - `sentenceCardLoadingHtml()`：句卡載入中先顯示對應 `.interactive-sentence-zh` 的既有譯文，再等 AI 詳解返回——句卡「秒開」。
- `validate.js`：`validateArticleAnalysis` 允許可選 `sentences:[{en,zh}]`。
- `ai-config.example.js`：新增 `AI_PROMPTS.article.paragraphSentences.user` 模板（本機 `ai-config.js` 缺此鍵時用內建預設）。

舊文章、舊快取無 `sentences` 欄位時自動回退原邏輯，完全相容。

## 工程修復：跨模型 JSON 穩定性（`modules/api.js`）

針對 reasoning 模型截斷殘骸問題，四項修復一次覆蓋文章詳解全部任務（點詞/句子/段落/片語）+ 例句/造句檢查：

### 1. 截斷自動重試（`requestAI`）
偵測 `finish_reason === 'length'`，自動把 `max_tokens` 加倍重試一次（上限 4000），用 `continue` 不佔用 429/5xx 的重試額度。這是「思考吃光額度」的直接解法。

### 2. 統一回應讀取（`pickMessageContent`）
順序 `content → reasoning_content → delta.content → output_text`，並支援 content 為陣列（合併文字片段）。原本 api.js 只讀 `message.content`，GLM/Qwen 那種把答案放錯欄位的模型直接拿空字串而失敗。

### 3. 拒收截斷殘骸
- 四個 analyze **主路徑**：JSON 解析失敗（`safeJsonParse` 回 null）即拋錯換模型/降級，主路徑 normalize 不再帶 `fallbackText`，杜絕把半截 JSON 塞進 `meaning`/`translation`。句子主路徑額外要求 `translation` 非空。
- **純文字後備路徑**：新增 `plainAnswerOrEmpty()`——回應以 `{`/`[` 開頭（截斷 JSON 殘骸）一律視為無效返回 `''`；後備產生空 meaning/translation 時拋錯，交由 UI 顯示「重試」。

**效果**：殘骸再也進不了顯示與快取（之前殘骸能通過 schema 驗證並被快取 14–30 天）。

### 4. 放寬 token 預算
| 任務 | 原 | 新 |
|---|---|---|
| 點詞 wordTooltip | 280 | 520 |
| 句子（含結構/精簡） | 650 / 380 | 1100 / 700 |
| 片語 | 260 | 520 |
| 段落（quick/standard） | 500 / 700 | 800 / 1000 |
| 段落按句模式 | — | `300 + 句數×180`（800–2400 夾逼） |
| 例句生成 | 600 | 900 |
| 造句檢查 | 400 | 600 |
| 各後備路徑 | 120–450 | 200–700 |

## 待辦 / 注意事項

1. **清一次 AI 快取**：先前 gpt-oss-20b 產生的殘骸已存進 IndexedDB（點詞 30 天 / 句子 21 天 / 段落 14 天 TTL）。不清的話再點同樣詞/句仍顯示舊殘骸。最快：DevTools → Application → IndexedDB → 刪 `bdc-cache`。
2. **fallback 模型建議分流**：目前 `ai-config.js` 把 article 主模型與 `*Fallback` 都設成 gpt-oss-20b，`buildModelCandidates` 去重後只剩 1 個候選，失敗時少了「換另一個模型」這層保護。建議把 `articleParagraphTranslationFallback` 等留一個不同模型（如 `bohe:gemini-2.5-flash`）。
3. **後續可選（尚未做）**：上一份調研的 P2/P3——提示詞統一「三段式」改寫、`response_format` 三級降級（json_schema→json_object→none）、提示詞分類重構（`modules/prompts.js` 唯一來源、任務鍵收斂）。
